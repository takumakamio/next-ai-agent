## 共通：プロジェクト全体で使う技術

### TypeScript — 安全な JavaScript

> **JavaScript に「型」を追加して、ミスを事前に防げるようにした言語**

フロントエンドもサーバーも、このプロジェクトでは **すべて TypeScript** で書きます。

#### たとえ話

JavaScript は「何でも入る箱」。TypeScript は「ラベル付きの箱」。
箱に「数値専用」とラベルが貼ってあれば、うっかり文字を入れようとしたときに **書いている段階で教えてくれます**。

#### 具体例

```
// JavaScript — 実行するまでエラーに気づかない
let age = "二十歳"    // 文字列を入れてもエラーにならない
age * 2              // 実行時に NaN（計算できない）になる

// TypeScript — 書いた瞬間にエラーを教えてくれる
let age: number = "二十歳"   // ここで即エラー！「文字列は number 型に入れられません」
```

#### なぜ TypeScript を使うのか

- **バグを事前に防げる** — 実行しなくてもミスがわかる
- **Claude Code との相性が良い** — 型情報があると AI がより正確なコードを生成できる
- **現在の主流** — 新しいプロジェクトのほとんどが TypeScript を採用

### npm — パッケージ管理ツール

> **他の人が作ったライブラリ（パッケージ）をインストール・管理するためのツール**

スマホの「App Store」のようなものです。自分で全部作らなくても、必要な機能を「パッケージ」としてダウンロードして使えます。

#### よく使うコマンド

| コマンド            | 意味                                   |
| ------------------- | -------------------------------------- |
| `npm install`       | package.json に書かれたパッケージを全部入れる |
| `npm install hono`  | hono パッケージを追加する                |
| `npm run dev`       | 開発サーバーを起動する                   |
| `npm run build`     | 本番用にビルドする                       |

> **npm は Node.js をインストールすると自動で付いてきます。** 別途インストールする必要はありません。

## フロントエンド：画面を作る技術

> フロントエンドとは、ユーザーが目にする **画面（UI）** の部分です。
> ブラウザの中で動きます。

### React — 画面を部品で組み立てるライブラリ

> **UI（ユーザーインターフェース）を「部品」の組み合わせで作るためのライブラリ**

#### たとえ話

レゴブロックのようなものです。「ボタン」「入力欄」「カード」といった小さな部品（**コンポーネント**）を作り、それを組み合わせて画面を構築します。

#### なぜ React を使うのか

- 部品ごとに分けて作れるので、**大きなアプリでも管理しやすい**
- 世界で最も使われている UI ライブラリなので、**情報が豊富**
- 一度作った部品を別の画面でも再利用できる

#### よく使う React の機能（Hooks）

React には **Hooks（フックス）** という仕組みがあり、コンポーネントに「状態」や「副作用」を持たせることができます。この研修で特によく使う 2 つを紹介します。

##### `useState` — コンポーネントの「状態」を管理する

> **画面に表示するデータを保存し、変更があれば自動で画面を更新してくれる仕組み**

```tsx
const [count, setCount] = useState(0)
//     ↑ 現在の値   ↑ 値を更新する関数    ↑ 初期値

// ボタンを押すたびに count が 1 増え、画面も自動で更新される
<button onClick={() => setCount(count + 1)}>
  {count} 回クリック
</button>
```

たとえ話：ホワイトボードに数字を書いておくイメージです。`setCount` で数字を書き換えると、それを見ている全員（画面）に自動で反映されます。

##### `useEffect` — 「タイミングを指定して処理を実行する」

> **コンポーネントが表示されたときや、特定の値が変わったときに処理を実行する仕組み**

```tsx
// ページが表示されたときに API からデータを取得する
useEffect(() => {
  fetch('/api/qas')
    .then(res => res.json())
    .then(data => setQas(data))
}, [])  // ← [] は「最初の1回だけ実行」という意味

// search の値が変わるたびにデータを再取得する
useEffect(() => {
  fetchData(search)
}, [search])  // ← search が変わるたびに実行
```

たとえ話：目覚まし時計のようなものです。「朝7時になったら（＝画面が表示されたら）コーヒーを淹れる（＝データを取得する）」というように、**タイミングと処理をセットで指定** します。

#### 他のフレームワークとの比較

| ライブラリ     | 特徴                                           | 普及度（2024-2025年調査）           |
| ------------- | ---------------------------------------------- | --------------------------------- |
| **React**     | 最も普及。自由度が高い。Meta（Facebook）が開発    | 1位（シェア約40〜45%で圧倒的）      |
| **Vue.js**    | 学習しやすい。日本やアジアで人気。テンプレート構文  | 2〜3位（調査により Angular と入替）  |
| **Angular**   | 大規模向け。Google が開発。企業利用が多い         | 2〜3位（調査により Vue と入替）     |
| **Svelte**    | 軽量・高速。コンパイル型で独自のアプローチ        | 利用者は約7%だが開発者満足度は1位   |

> **この研修で React を選んだ理由：** 最も普及しており、Claude Code が最も得意とするライブラリだから

### Next.js — React をもっと便利にするフレームワーク

> **React だけでは面倒なこと（ルーティング、サーバー処理、パフォーマンス最適化）を全部やってくれるフレームワーク**

#### React だけだと困ること → Next.js が解決

| 困ること                           | Next.js の解決策                     |
| ---------------------------------- | ----------------------------------- |
| ページ遷移の仕組みを自分で作る必要   | ファイルを置くだけで自動ルーティング   |
| サーバー側の処理を別で用意する必要   | API ルートを同じプロジェクト内に書ける |
| 初回表示が遅い                      | サーバーサイドレンダリング（SSR）     |
| 画像の最適化を自分でやる必要         | 自動で最適化してくれる               |

#### たとえ話

React が「木材・釘・ペンキ」だとすると、Next.js は「設計図付きの家キット」です。素材は同じですが、組み立て方のガイドが付いています。

#### 他のフレームワークとの比較

| フレームワーク | ベース    | 特徴                              |
| ------------- | --------- | -------------------------------- |
| **Next.js**   | React     | 最も人気。Vercel が開発。フルスタック |
| **Remix**     | React     | データ取得が得意。Web標準重視       |
| **Nuxt.js**   | Vue.js    | Vue 版の Next.js 的存在            |
| **SvelteKit** | Svelte    | Svelte 版のフルスタックフレームワーク |

#### たとえ話

料理で言えば「半調理済みの食材キット」。ゼロから作らなくても、プロ品質の UI パーツが揃っていて、自分のプロジェクトに合わせてカスタマイズできます。

#### shadcn/ui で使えるパーツの例

```
ボタン / 入力欄 / テーブル / ダイアログ（ポップアップ）/

```

| ライブラリ     | 特徴                                              |
| ------------- | ------------------------------------------------- |
| **shadcn/ui** | コードをコピーして使う。カスタマイズしやすい。最近主流   |

## サーバーサイド：裏側の処理を作る技術

> サーバーサイドとは、ユーザーには見えない **裏側の処理** です。
> データの保存、AI への問い合わせ、ビジネスロジックなどを担当します。

### Hono — 軽量な API サーバー

> **サーバー側の処理（API）を書くための軽量・高速なフレームワーク**

#### API とは？

レストランの「注文窓口」のようなものです。

```
お客さん（ブラウザ）   →  「カレーください」（リクエスト）  →  厨房（サーバー）
お客さん（ブラウザ）   ←  カレー（レスポンス）             ←  厨房（サーバー）
```

Web アプリでは：
- 「Q&A の一覧をください」→ データベースから取得して返す
- 「この質問に答えて」→ AI に聞いて回答を返す

#### 他のフレームワークとの比較

| フレームワーク | 特徴                                         |
| ------------- | -------------------------------------------- |
| **Hono**      | 超軽量・高速。TypeScript ファースト。最近人気急上昇 |
| **Express**   | 最も歴史がある。情報は多いが設計が古い           |
| **Fastify**   | Express の後継的。プラグインシステムが充実        |
| **tRPC**      | API を型安全に呼び出せる。フルスタック TypeScript 向け |

> **この研修で Hono を選んだ理由：** 軽量で、Next.js の中に組み込みやすく、TypeScript との相性が抜群だから

### PostgreSQL — データを保存するデータベース

> **データを整理して保存・検索できる「データの倉庫」**

#### たとえ話

Excel のスプレッドシートを想像してください。行と列があり、データを整理して保存します。データベースはその「超高性能版」で、大量のデータを高速に検索・更新できます。

#### 他のデータベースとの比較

| データベース     | 特徴                                            |
| --------------- | ----------------------------------------------- |
| **PostgreSQL**  | 高機能・信頼性が高い。拡張機能（pgvector など）が豊富 |
| **MySQL**       | シンプルで高速。Web アプリで広く使われている         |
| **SQLite**      | ファイル1つで動く。小規模アプリ向け                  |
| **MongoDB**     | JSON 形式でデータを保存。柔軟だが複雑なクエリが苦手  |

> **この研修で PostgreSQL を選んだ理由：** pgvector 拡張で **ベクトル検索**（AI 機能に必須）が使えるから

#### pgvector とは？

PostgreSQL の拡張機能で、**テキストの「意味の近さ」を計算して検索** できます。

```
通常の検索：   「TypeScript」で検索 → 「TypeScript」を含むデータだけヒット
ベクトル検索：  「TypeScript」で検索 → 「型安全な JavaScript」もヒット（意味が近いから）
```

### Drizzle ORM — データベースを簡単に操作するツール

> **SQL（データベースの言語）を書かなくても、TypeScript のコードでデータベースを操作できるツール**

#### たとえ話

外国のレストランで注文するとき、現地語（SQL）を話せなくても、翻訳アプリ（ORM）を使えば注文できる。ORM は TypeScript とデータベースの間の「翻訳機」です。

#### 具体例

```
-- SQL（データベースの言語）
SELECT * FROM qas WHERE category = 'programming' ORDER BY created_at DESC;

// Drizzle ORM（TypeScript で書ける）
db.select().from(qas).where(eq(qas.category, 'programming')).orderBy(desc(qas.createdAt));
```

#### 他の ORM との比較

| ORM           | 特徴                                         |
| ------------- | -------------------------------------------- |
| **Drizzle**   | 軽量。SQL に近い書き方。型安全。最近人気急上昇    |
| **Prisma**    | 最も人気。独自のスキーマ言語を使う               |
| **TypeORM**   | デコレーターベース。歴史がある                   |
| **Kysely**    | SQL ビルダー。ORM というよりクエリビルダー        |

> **この研修で Drizzle を選んだ理由：** SQL の書き方に近く直感的、TypeScript との型連携が優秀だから

### Gemini AI — Google の AI モデル

> **Google が提供する AI。質問に回答したり、テキストをベクトル（数値の配列）に変換したりできる**

#### この研修で使う 2 つの機能

| 機能         | 使い方                                                  |
| ------------ | ------------------------------------------------------- |
| **会話**     | ユーザーの質問に対して、Q&A データを参考にしながら回答を生成 |
| **Embedding** | テキストを 2000 個の数値に変換して、意味の近さを計算する   |

#### Embedding（埋め込み）とは？

テキストを数値の配列に変換する技術です。意味が似た文章は、似た数値の並びになります。

```
「TypeScript とは？」    →  [0.12, 0.85, 0.33, 0.67, ...]
「型安全な JavaScript」 →  [0.15, 0.82, 0.31, 0.69, ...]  ← 数値が似ている！（意味が近い）
「今日の天気」           →  [0.91, 0.04, 0.78, 0.12, ...]  ← 数値が全然違う（意味が遠い）
```

#### 他の AI モデルとの比較

| AI モデル      | 提供元      | 特徴                                |
| -------------- | ---------- | ----------------------------------- |
| **Gemini**     | Google     | マルチモーダル。Embedding API が充実   |
| **GPT**        | OpenAI     | 最も有名。ChatGPT のベース            |
| **Claude**     | Anthropic  | 長文理解に強い。安全性重視             |
| **LLaMA**      | Meta       | オープンソース。自前サーバーで動かせる  |

> **この研修で Gemini を選んだ理由：** 無料枠が大きく、Embedding API が使いやすく、研修で試しやすいから

### サーバーサイドまとめ

```

```

## セキュリティ：アプリを安全に作るために

> Web アプリを作るとき、**セキュリティ（安全性）** は避けて通れません。
> ここでは初心者が最低限知っておくべきポイントを紹介します。
> 難しく考える必要はありません。この研修で使う技術には、**セキュリティを助ける仕組みが最初から入っている** ものが多いです。

### API キーの管理 — 一番大事なこと

API キー（Gemini AI を使うためのパスワードのようなもの）は、**絶対に他人に見せてはいけません**。

```
やってはいけないこと：
  ❌ API キーをコードに直接書く
  ❌ API キーを GitHub にアップロードする
  ❌ API キーを他人にチャットで送る

正しい管理方法：
  ✅ .env.local ファイルに書く（このファイルは Git にアップされない）
  ✅ .gitignore に .env.local が含まれていることを確認する
```

```
# .env.local（このファイルは自分の PC にだけ存在する）
GOOGLE_GENERATIVE_AI_API_KEY=AIzaSy...（ここに自分のキー）
```

> **なぜ危険？** API キーが漏れると、他人があなたのキーを使って API を大量に呼び出し、**高額な請求が発生する** 可能性があります。

### 入力バリデーション — 不正なデータを受け付けない

ユーザーからのデータは **常に疑ってかかる** のが鉄則です。

```
悪意のある例：
  ユーザーが question 欄に
  「'; DROP TABLE qas; --」と入力 → データベースが消える可能性！（SQL インジェクション）

対策：
  ✅ Zod でリクエストの形式をチェック（バリデーション）
  ✅ Drizzle ORM を使う（SQL を直接書かないので安全）
```

この研修では **Zod**（バリデーションライブラリ）と **Drizzle ORM** を使うので、主要な攻撃は自動的に防がれます。

### CSRF 対策 — なりすましリクエストを防ぐ

> **CSRF（クロスサイトリクエストフォージェリ）** とは、悪意のあるサイトからユーザーになりすまして API を呼び出す攻撃です。

```
攻撃の流れ：
  1. ユーザーがあなたのアプリにログイン中
  2. 別のタブで悪意のあるサイトを開く
  3. そのサイトが裏であなたのアプリの API を勝手に呼び出す
  → ユーザーの意図しない操作が実行される

対策：
  ✅ Hono の CSRF ミドルウェアを有効にする（この研修で設定します）
```

### CORS 設定 — アクセス元を制限する

> **CORS（クロスオリジンリソースシェアリング）** とは、「どのサイトからの API アクセスを許可するか」を制御する仕組みです。

```
CORS なし：
  どのサイトからでも API を呼び出せてしまう → 危険

CORS あり：
  自分のアプリ（localhost:3000）からだけ許可 → 安全
```

### この研修で使うセキュリティ機能まとめ

| 脅威                       | 対策                    | 使う技術         |
| -------------------------- | ----------------------- | --------------- |
| API キーの漏洩              | `.env.local` で管理     | dotenv          |
| 不正なデータ入力            | リクエストのバリデーション | **Zod**         |
| SQL インジェクション        | ORM 経由の DB 操作       | **Drizzle ORM** |
| なりすましリクエスト（CSRF） | CSRF トークン検証        | **Hono**        |
| 不正なアクセス元（CORS）    | オリジン制限             | **Hono**        |

> **安心ポイント：** この研修で使う技術（Zod、Drizzle ORM、Hono）は、セキュリティ機能が組み込まれています。Claude Code に「セキュリティ対策もお願い」と伝えれば、適切な設定を入れてくれます。
